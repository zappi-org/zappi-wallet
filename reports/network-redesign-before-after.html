필요한 것만 한 번씩 묻게 만들기

민트·릴레이 서버가 “너무 자주 물어본다”며 차단(rate limit)하던 문제를, 7단계에 걸쳐 구조적으로 고쳤습니다. 각 단계마다 기존(전) → 변경(후) → 왜를 그림으로 설명합니다.

0. 등장인물 1. 관문·버그수술 2. 책갈피(cursor) 3. 민트 정보 창구 4. 복구 해체 5. stuck-sweep 6. 릴레이 교환원 7. 견적·타이핑 8. 안전장치
0

등장인물과 문제의 정체

비유: 민트 = 은행 지점(토큰 검증·결제 확인·수수료), 릴레이 = 우체국(다른 사람이 보낸 ecash 편지가 쌓이는 곳). 서버에 묻는 방법은 폴링(내가 주기적으로 전화)과 푸시(소식 오면 서버가 알려줌) 두 가지.
앱 사용 1시간 동안의 서버 요청 밀도 (개념도)
개편 전: 켜기만 해도 전체 재다운로드, 화면마다 전수 확인, 30초마다 전건 폴링
개편 후: 푸시가 기본, 폴링은 “막힌 것만”, 같은 질문은 한 번만
1

요청 관문(RequestGate) + 잠복 버그 수술

비유: 엘리베이터 버튼을 다섯 명이 연달아 눌러도 엘리베이터는 한 대만 옵니다. 같은 요청이 여러 곳에서 동시에 와도 실제 요청은 하나만 나가고, 나머지는 그 결과를 나눠 씁니다.
기존

잠금 해제·화면 진입·새로고침이 각자 같은 복구/확인 요청을 쐈습니다. 5군데가 부르면 5번 나감.

잠금해제홈 진입새로고침 →→→요청 ×3

그리고 30초 폴링이라는 “보험”이 진짜 버그들을 가려주고 있었습니다: 송금 실패가 화면에 안 뜨고 잠복, 비행기 모드로 켜면 푸시 수신기가 세션 내내 꺼짐, 백그라운드에서도 폴링 계속.

변경

모든 고빈도 요청 앞에 관문을 세웠습니다: ① 지금 진행 중이면 그 결과를 같이 기다림(in-flight 공유) ② 방금 성공했으면 잠시(쿨다운) 재실행 안 함.

잠금해제홈 진입새로고침 →🚪 관문 →요청 ×1

가려져 있던 버그도 수술: 송금 실패 즉시 표시, 온라인 복귀 시 수신기 재가동, 백그라운드 폴링 정지. 그리고 계측기(카운터)를 달아 이후 단계의 효과를 숫자로 채점할 준비를 했습니다.

왜: 보험을 걷어내려면(5단계) 보험이 가려주던 병부터 고쳐야 하고, 고쳤다는 걸 증명할 계측기가 먼저 필요했습니다.
2

릴레이에 책갈피 만들기(cursor)

비유: 500권짜리 전집을 읽다가 덮은 뒤, 책갈피가 없으면 다시 펼 때마다 1권 1쪽부터. 책갈피가 있으면 거기서부터. 단, 이 편지들은 프라이버시를 위해 발송 시각이 최대 2일 과거로 적혀 있어서, 책갈피보다 2일+6시간 앞에서부터 겹쳐 읽습니다(중복은 “처리한 편지 목록”이 걸러냄).
기존

앱을 켤 때마다, 연결이 잠깐 끊겼다 붙을 때마다 편지함 전체 역사를 다시 다운로드. 릴레이 rate limit의 최대 원인.

변경

“이 릴레이에게 언제까지 전부 받았다”를 디스크에 기록하고, 다음엔 그 이후 것만 요청.

안전 규칙: 책갈피는 릴레이가 “여기까지가 전부”라는 진짜 신호(EOSE)를 줬을 때만 전진. 시간이 초과돼 끊긴 것(timeout)은 절대 전진시키지 않음. 놓친 편지는 곧 돈이기 때문입니다.

왜: 재다운로드는 릴레이 입장에서 “수천 통을 매번 다시 달라”는 요청입니다. 책갈피 하나로 요청량이 역사 전체 → 마지막 확인 이후로 줄어듭니다.
3

민트 정보 창구 단일화(MintInfoService)

비유: 회사에서 5개 팀이 각자 은행에 전화해 “지점 영업하나요?”를 물었습니다. 이제는 안내 데스크 한 곳이 대표로 묻고, 모두가 그 답을 공유합니다.
기존

같은 민트에 같은 질문이 화면 수만큼 중복 발사.

변경

받은 정보(이름·아이콘)는 24시간 캐시에 저장되어, 상세 화면은 아예 네트워크 없이 그립니다.

왜: 정보의 주인이 한 명이면 중복이 구조적으로 불가능해집니다. “누가 이 요청의 주인인가”가 이 개편 전체의 반복 주제입니다.
4

“전부 다시 확인” 해체(Recovery 재편)

비유: 집을 나설 때마다 가스·수도·창문·자물쇠·보일러를 전부 점검하던 사람이, “외출 = 자물쇠만, 장기 여행 = 전부”로 상황에 맞게 점검하게 됐습니다.
기존

recoverAll 함수 하나가 9가지 일(네트워크 전수 질의 + 로컬 장부 정리가 뒤섞임)을 몽땅 했고, 이걸 6개 트리거가 각자 불렀습니다:

네트워크 sweep멜트 재확인장부 settle레거시 회수만료 정리원격 quote 확인stuck 강제실행행 청소오프라인 상환
잠금해제●●●●●●●●●
앱 복귀●●●●●●●●●
당겨서 새로고침●●●●●●●●●
토큰 탭 진입●●●●●●●●●
민트 추가●●●●●●●●●
설정 복구 버튼●●●●●●●●●

붉은 ● = 실행. 탭 하나 열었을 뿐인데 민트 전수 질의가 나가던 이유.

9가지 일을 성격별로 3개 경로로 나누고, 트리거마다 필요한 것만:

reconcile
(로컬 장부 정리, 네트워크 0)		recoverTargeted
(푸시가 못 닿는 것만, 5분 관문)		runFullNetworkRecovery
(전수 점검)
잠금해제 / 복귀 / 새로고침●● (관문 통과 시)·
토큰 탭 진입●··
민트 추가(신뢰)·● + 대기열 상환·
설정 복구 버튼●●● (사용자 명시 의도)
이 과정에서 발견한 돈 잃는 버그: “모르는 민트에서 온 토큰” 확인 대기열이 포스트잇(메모리)이었고, 도장 찍는 순서가 반대였습니다.
기존 (유실 경로)
  • 1편지 도착 → “처리했음” 도장부터 디스크에 찍음
  • 2확인 대기열(포스트잇)에 붙임 (메모리에만)
  • 3💥 여기서 새로고침/크래시
  • 4포스트잇 증발. 도장은 남아서 재수신도 차단토큰 영구 유실
변경 (순서 교정)
  • 1편지 도착 → 대기열을 디스크(노트)에 먼저 기록
  • 2그 다음에 “처리했음” 도장
  • 3💥 사이에 죽어도: 도장이 없음
  • 4다음 실행 때 같은 편지 재수신 → 재기록(중복 안전)유실 불가

부팅 시 노트를 읽어 확인 창을 다시 띄우고, 민트를 신뢰하는 순간 그 민트의 대기 토큰을 자동 상환합니다(“복구액” 표시도 이 실제 합계로).

왜: 돈이 걸린 코드에서 “디스크에 먼저, 도장은 나중에” 순서 하나가 유실/무유실을 가릅니다. 그리고 로컬 장부 정리는 네트워크가 필요 없다는 걸 분리해내면, 고빈도 트리거(탭 진입)가 공짜가 됩니다.
5

30초 전수 폴링 → 120초 “막힌 것만”(stuck-sweep)

비유: 택배 5개를 기다리며 30초마다 5통씩 배송조회 전화를 하던 사람이, 이제는 2분마다 문 앞(디스크)만 확인하고, “2분 넘게 소식 없는 택배”만 골라 전화합니다. 기다리는 택배가 없으면 알람 자체를 끕니다.
기존
진행 중 5건 기준, 10분간 원격 요청
30초 × 5건 = 10분에 100회 원격 확인 (푸시가 멀쩡해도)

백그라운드에서도 계속. “푸시를 못 믿으니 전부 다시 물어보자”의 전형.

변경
2분 타이머→ ① 디스크만 확인 (전화 0통)→ ② 2분째 무소식인 것만→ 원격 확인 1회
푸시가 정상이면 ②까지 갈 일이 거의 없음 → 원격 요청 0~수 회

진행 건 0이면 타이머 정지, 새 송금이 생기면(다른 탭 포함) 자동 재개. 원격 확인이 실패하면 상태를 건드리지 않고 다음 주기에 재시도합니다. 일시적인 민트 다운이 진행 중인 결제를 “실패”로 확정하면 안 되기 때문입니다.

왜 + 채점: 이 강등이 안전한지 확인할 지표를 함께 만들었습니다: tls_stuck_detected(sweep이 푸시 대신 구해준 건수)가 7일간 0이면 푸시가 제 역할을 다 한다는 증거. 리뷰에서 이 지표가 두 번 뒤집혔습니다: 처음엔 “사용자가 그냥 기다리는 것”까지 세서 항상 낙제 → 좁혔더니 이번엔 수신 푸시가 죽어도 만점이 나오는 구멍 → 최종적으로 “원격이 내 디스크보다 앞서 있던, 만료 탓이 아닌 변화”만 세도록 확정.
6

릴레이 연결 교환원(NostrSessionController)

비유: 예전엔 전화선 관리가 중구난방이라, 늦게 연결된 우체국은 구독 신청이 누락되고, DM 한 통 보내면 내 전화번호부가 상대방 것으로 바뀌는 사고까지 있었습니다. 이제 교환원 한 명이 회선과 구독 명단을 책임집니다.
기존: 결함 3가지
  • 1구독이 “그 순간 연결된” 릴레이에만 부착. 늦게 붙은 릴레이의 편지는 영영 못 받음
  • 2릴레이 1개 재연결 → 전체 구독을 전부 껐다 켬 (낭비 + 재다운로드)
  • 3DM 발송 시 connect(상대방 릴레이)가 내 기본 연결 목록을 통째로 교체. 이후 앱이 상대 우체국만 바라봄
변경: 교환원의 3원칙
  • 1명단제: 구독은 명단에 등록되고, 릴레이가 (재)연결되는 순간 자동 부착. 구독 끊김은 구독 자신의 종료 신호(onclose)로 감지
  • 2그 릴레이만 다시 부착. 나머지는 건드리지 않음
  • 3임대(lease): 상대방 릴레이는 “빌린 회선”. 보내고 확인되면 2분 뒤 자동 반납. 내 기본 목록 불변
리뷰가 잡은 치명타

새 구조에서 “내 기본 릴레이에 연결하라”는 명령을 아무도 안 내리고 있었습니다. 예전 코드는 DM 함수 안에 숨은 부수효과가 우연히 연결을 만들어줬는데, 새 구조는 그 길을 지나가지 않았고, 테스트 944개가 전부 통과한 채 실시간 수신·발행이 죽는 상태였습니다.

수정: 잠금 해제 때 명시적으로 연결 + 이 배선이 사라지면 CI가 실패하는 전용 테스트 추가. 교훈: “우연히 되던 것”은 명시적 코드로 승격해야 한다.

왜: 연결·구독·재연결의 주인이 한 명이면, “누가 언제 붙였지?”류의 레이스가 구조적으로 사라집니다. 3단계(창구), 4단계(복구 주인)와 같은 원리를 릴레이 연결에 적용한 것.
7

마무리: 수수료 이중 견적 제거, 타이핑 중 네트워크 금지

기존 ① 이중 견적
금액 입력 화면→견적 (민트 4왕복)→ 확인 화면→그 결과를 버리고 재견적 4왕복

내 지갑 간 이체 확인 화면 한 번에 최대 8왕복.

변경 ①
금액 입력 화면→견적 4왕복→ 확인 화면→그 값을 그대로 신뢰 (0왕복)

같은 함수·같은 인자임을 리뷰에서 검증. 같은 조합의 견적은 60초 재사용, 토큰 탭의 회수 수수료도 한 번 계산한 건 캐시.

기존 ② 타이핑 중 검증

a@gmail.com을 치는 도중 0.5초 멈추면:

a@gmail.co→gmail.co 에 실제 GET 요청 발사

반쯤 입력된 주소마다 엉뚱한 서버로 요청이 나감.

변경 ②
타이핑 중→모양 검사만 (이메일처럼 생겼나?)
“다음” 버튼→그때 딱 한 번 실제 검증

붙여넣기·QR 스캔은 “입력이 완성된 순간”이므로 기존대로 즉시 검증.

8

안전장치: 되돌림 스위치와 채점판

이 모든 변경에는 즉시 예전 방식으로 돌아가는 스위치가 붙어 있습니다. 브라우저 localStorage에 값 1을 넣고 잠금을 다시 풀면 그 단계만 구버전 동작으로 복귀합니다.

zappi.ks.cursor
켜면: 책갈피 무시, 전체 재다운로드(2단계 이전)
zappi.ks.mint-info-facade
켜면: 민트 정보 창구 해체, 화면별 개별 요청(3단계 이전)
zappi.ks.recovery-split
켜면: recoverAll이 다시 9가지 전부 실행(4단계 이전)
zappi.ks.tls-sweep
켜면: 30초 전수 폴링 복귀(5단계 이전)
zappi.ks.nostr-controller
켜면: 교환원 해임, 구 연결 방식(6단계 이전)

채점판은 앱 안에 있습니다: 설정 → 환경설정 → 진단. 5단계(폴링 강등)의 합격 기준은 아래 두 숫자로 판정합니다(팀 기기 3대, 7일간):

tls_stuck_detected
= 0
sweep이 푸시 대신 구해준 건수. 0이어야 “푸시가 제 역할을 다 한다”는 증거
coco_push_received
