Coco 통합 및 Rate Limit 분석 보고서

Zappi Wallet에서 민트/릴레이 rate limit이 자주 발생하는 원인을 Coco가 이미 제공하는 기능, Zappi가 별도로 구현한 기능, 통합 가능한 기능으로 나누어 정리한 의사결정용 보고서입니다.

1. 결론 요약

핵심 판단

민트 네트워크 요청의 주 소유자는 Coco로 맞추는 것이 맞습니다. Coco는 이미 mint별 rate limiter, mint WSS + fallback polling, operation recovery, proof state watcher를 갖고 있습니다.

가장 큰 중복

Zappi는 Coco 초기화 직후에도 자체 recoverAll(), 30초 transfer polling, 직접 /v1/info 조회, 직접 cashu-ts 복구를 추가로 수행합니다.

분리해야 할 영역

Nostr 릴레이, NIP-17 gift wrap, 프로필, 고객지원 릴레이는 Coco의 범위가 아닙니다. 이 부분은 Zappi가 계속 소유하되 relay scope와 query limit을 고쳐야 합니다.

가장 먼저 줄일 수 있는 rate limit 원인

직접 fetch(.../v1/info)와 직접 cashu-ts Mint/Wallet 사용은 Coco의 MintRequestProvider rate limiter를 우회합니다. 이 요청들을 Coco 경유 또는 단일 Zappi mint-info facade로 모아야 합니다.

중요한 구조적 중복

initializeCoco()는 초기화 과정에서 이미 send/melt/receive/mint recovery를 실행합니다. 그런데 Zappi는 unlock 직후 activate() 후 다시 refreshAndRecover()를 실행하고, resume, pull-to-refresh, Token 탭, Add Mint에서도 recoverAll()을 다시 호출합니다.

권장 방향

민트 요청은 Coco 중심으로 통합하고, Zappi는 앱 도메인 상태 관리와 Nostr transport에 집중하는 구조가 가장 단순합니다. 즉, Coco = mint wallet engine, Zappi = product workflow + Nostr + transaction UI로 책임을 분리합니다.

2. Coco가 지원하는 것

영역 Coco 지원 내용 Zappi에 주는 의미
Mint 요청 제한 MintRequestProvider가 mint별 token bucket rate limiter를 제공합니다. 기본값은 capacity: 20, refillPerMinute: 20입니다. Coco 경유 mint 요청은 중앙 limiter를 탑니다. 반대로 Zappi의 직접 fetch와 직접 cashu-ts 호출은 이 보호 밖에 있습니다.
Mint 정보/키셋 manager.mint.addMint(), getMintInfo(), trust/untrust, keyset fetch/update를 제공합니다. mint 추가, mint metadata, health/info 조회를 가능하면 한 경로로 합쳐야 합니다.
Wallet 기능 manager.wallet.receive(), restore(), sweep(mintUrl, bip39seed), balances, token decode/encode를 제공합니다. 현재 지갑 복구는 이미 wallet.restore()를 사용합니다. 다른 복구문구에서 이캐시 가져오기는 wallet.sweep()로 대체 가능성이 큽니다.
Operation API ops.send, ops.receive, ops.mint, ops.melt가 prepare/execute/refresh/cancel/reclaim/finalize/recovery/diagnostics를 제공합니다. Zappi의 Cashu adapter는 대부분 이미 이 API를 감싸고 있습니다. 중복은 adapter 주변의 재복구/polling 레이어에서 발생합니다.
Recovery initializeCoco()가 초기화 시 send, melt, receive recovery와 pending mint operation recovery를 실행합니다. 앱 unlock 직후 별도 recoverAll()은 네트워크 복구 관점에서는 중복일 가능성이 큽니다. 다만 Zappi transaction DB 정합성 보정은 따로 남길 수 있습니다.
Watcher enableMintOperationWatcher(), enableProofStateWatcher(), enableMintOperationProcessor()를 제공합니다. Lightning receive, ecash send claim 상태는 이벤트 기반으로 받을 수 있습니다. 별도 30초 polling은 fallback으로만 제한하는 편이 좋습니다.
Mint WSS + polling SubscriptionManager는 브라우저 WebSocket이 있으면 /v1/ws WSS와 polling을 병행합니다. 기본은 WSS 연결 중 20초 slow polling, WSS 실패 시 5초 fast polling입니다. Coco watcher 자체도 polling을 포함합니다. Zappi의 transfer polling과 합쳐지면 같은 quote/proof를 여러 경로로 확인할 수 있습니다.
Nostr relay Coco의 WSS는 Cashu mint /v1/ws용입니다. Nostr relay WSS, NIP-17, profile, support는 담당하지 않습니다. 릴레이 rate limit은 Coco 통합으로 해결되지 않습니다. Zappi Nostr 계층에서 별도로 고쳐야 합니다.

3. 현재 Zappi 구조

3.1 정상적으로 Coco를 사용 중인 부분

토큰 보내기 prepareSend()와 executeSend()가 manager.ops.send.prepare/execute를 사용합니다. P2PK target도 Coco send target으로 넘깁니다.
토큰 받기 receiveToken()이 manager.ops.receive.prepare/execute를 사용합니다. 수수료 계산 후 execute하는 구조입니다.
Lightning 받기 createMintQuote(), redeemMintQuote(), checkMintQuote()가 manager.ops.mint를 사용합니다.
Lightning 보내기 prepareMelt(), executeMelt(), checkMelt(), rollbackMelt()가 manager.ops.melt를 사용합니다.
현재 지갑 복구 설정의 현재 지갑 잔액 확인은 mint별 manager.wallet.restore(mintUrl)를 호출합니다. 이 자체는 Coco 경유이므로 방향은 맞습니다.

3.2 Coco 위에 Zappi가 추가로 얹은 레이어

레이어 역할 판단
Coco event bridge Coco event를 Zappi EventBus, Zustand store, transaction DB에 연결 필요 앱 도메인 상태 반영용입니다.
TransferLifecycleService 프로토콜 중립 transfer 상태 머신과 30초 fallback polling 축소 후보 Coco event로 대부분 해결되면 polling은 더 긴 fallback으로 낮출 수 있습니다.
PaymentService.recoverAll() adapter별 pending melt/quote/send/offline receive 복구 중복 후보 Coco init recovery와 겹칩니다. local DB reconciliation과 network recovery를 분리해야 합니다.
Mint health/metadata 화면 진입마다 /v1/info 조회 통합 필요 Coco limiter 밖의 직접 fetch가 많습니다.
NostrGateway NIP-17 gift wrap, profile, Nostr direct payment, recovery anchor 계속 Zappi 소유 Coco 범위가 아닙니다.
Customer support relay nostr-cs 기반 별도 support WSS/client 계속 Zappi 소유 다만 connect/refresh single-flight는 필요합니다.

4. 불필요하거나 중복 가능성이 큰 것

4.1 확신도 높음

항목 현재 동작 왜 문제인가 권장 변경
직접 /v1/info fetch Home, MintManagement, MintInfoSheet, Add Trusted Mint, metadata service가 각각 mint info를 조회합니다. Coco MintRequestProvider의 per-mint limiter를 우회하고, health와 metadata가 같은 endpoint를 따로 칩니다. 하나의 mint-info facade로 통합하고, 가능하면 manager.mint.getMintInfo() 또는 Coco 경유 API를 사용합니다.
unlock 직후 recoverAll initializeCoco()가 recovery를 실행한 뒤, Zappi가 activate() 후 다시 refreshAndRecover()를 실행합니다. 같은 pending operation을 짧은 시간에 중복 조회할 수 있습니다. unlock 직후 recoverAll()은 제거하거나, Coco init recovery 이후 최소 cooldown/single-flight를 둡니다.
반복 recoverAll trigger resume, pull-to-refresh, Token 탭, Add Mint, Settings 복구에서 반복 호출됩니다. 사용자 행동과 visibility 이벤트가 겹치면 mint 복구 요청이 몰립니다. 전역 recovery scheduler를 만들고, network recovery와 local reconciliation을 분리합니다.
외부 복구문구 직접 cashu-ts 복구 Mint, Wallet, getKeySets(), loadMint(), batchRestore(), checkProofsStates()를 직접 호출합니다. Coco limiter, wallet cache, repository 흐름을 우회합니다. keyset 수가 많은 mint에서 요청 수가 커집니다. Coco의 manager.wallet.sweep(mintUrl, bip39seed)로 대체 가능한지 검증합니다. 어렵다면 최소한 요청 제한/취소/순차화를 추가합니다.
NUT-18 HTTP poller expiry 전달 누락 Receive 화면은 expiresAt을 넘기지만 bootstrap wrapper가 poller에 전달하지 않습니다. 요청 만료 후에도 max duration까지 endpoint polling이 이어질 수 있습니다. bootstrap.ts에서 expiresAt: opts.expiresAt을 전달합니다.

4.2 중간 확신도, 검증 후 축소

항목 현재 동작 검증할 것 권장 방향
30초 transfer polling Coco event bridge가 있음에도 active transfer를 30초마다 operator별로 poll합니다. Coco watcher/event가 모든 정상 케이스를 커버하는지, 앱 background/resume에서 누락이 실제로 있는지 확인해야 합니다. 기본은 event 기반으로 두고, polling은 2분 이상 장주기 fallback 또는 stuck transfer만 대상으로 제한합니다.
cleanAndRecoverStaleMintOps Coco 내부 pending mint ops를 직접 listPending() 후 mintAndReceive() 또는 abandon합니다. Coco의 recoverPendingMintOperations(), requeuePaidMintQuotes()와 중복되는지 확인이 필요합니다. SDK recovery가 충분하면 제거하고, 필요한 경우 transaction DB 보정만 남깁니다.
checkProofStates 직접 cashu-ts ecash send polling에서 token proof 상태를 직접 mint에 조회합니다. Coco proof state watcher가 해당 operation finalization을 충분히 이벤트로 주는지 확인해야 합니다. event 기반으로 줄이고, 직접 proof check는 manual reclaim/detail 화면에만 제한합니다.

4.3 계속 유지해야 하는 것

항목 유지 이유 개선 포인트
NostrIncomingWatcher Coco는 NIP-17 gift wrap 수신을 담당하지 않습니다. since, limit, processed cursor를 넣어 과거 이벤트 전체 조회를 피합니다.
Recovery anchor Zappi의 Nostr 기반 상태 복구 개념입니다. Coco Cashu wallet recovery와 범위가 다릅니다. fetchGiftWraps()에 limit/since를 추가하고 anchor 조회와 reconstruction의 중복 fetch를 줄입니다.
Nostr direct payment/profile kind 10019/10050, profile publish는 Coco 범위 밖입니다. connectedRelays 누적 문제를 줄이기 위해 query/publish 대상 relay를 명시적으로 스코프화합니다.
Customer support nostr-cs 기반 별도 기능입니다. connect() 중복 호출을 single-flight 처리하고, page/global refresh를 합칩니다.

5. 통합 전략

원칙 1
Mint network owner는 Coco로 둡니다.

Cashu mint HTTP, mint WSS, quote/proof 상태 추적, operation recovery는 Coco가 이미 가진 책임입니다. Zappi가 같은 mint endpoint를 직접 호출해야 한다면 반드시 이유를 남기고 단일 제한기를 거쳐야 합니다.

원칙 2
Zappi는 app workflow owner로 남깁니다.

거래내역 생성, toast, 화면 상태, pending review, Nostr gift wrap, support는 Coco가 모르는 제품 레이어입니다. 이 레이어는 Coco event를 받아 app domain event로 변환하는 방식이 맞습니다.

원칙 3
복구는 하나의 scheduler만 네트워크를 소유하게 합니다.

recovery API가 여러 화면과 lifecycle에서 직접 호출되면 사용자가 한 번 앱을 열었을 뿐인데 같은 mint에 여러 묶음 요청이 갑니다. 전역 single-flight, cooldown, task type 분리가 필요합니다.

원칙 4
Nostr relay는 Coco와 별개로 최적화합니다.

NostrGateway는 persistent relay와 temporary relay를 구분하고, query/publish 함수가 대상 relay를 명시적으로 받도록 고쳐야 합니다.

권장 목표 구조

책임 소유자 구현 방향
Cashu mint HTTP/WSS Coco Coco API 또는 Coco 기반 adapter만 사용합니다.
Mint info/cache/health Coco + Zappi facade UI용 상태는 Zappi facade가 관리하되 network fetch는 Coco 경유로 통합합니다.
Operation recovery Coco Zappi는 recovery 결과를 transaction DB에 반영하는 reconciliation만 담당합니다.
Transaction DB/UI 상태 Zappi Coco event를 받아 idempotent하게 settle/update합니다.
Nostr relay/gift wrap/profile/support Zappi since/limit, scoped relay pool, single-flight connect를 적용합니다.

6. 실행 우선순위

순위 작업 효과 위험도
P0 NUT-18 HTTP poller에 expiresAt 전달 만료 후 불필요한 HTTP polling 제거 낮음
P0 recoverAll(), checkAllMints(), support connect() single-flight/cooldown 중복 클릭/visibility/화면 진입으로 인한 요청 폭주 완화 낮음
P1 직접 /v1/info 호출을 단일 mint-info facade로 통합 health/metadata/add/detail 중복 요청 제거 중간
P1 unlock 직후 recoverAll() 제거 또는 local reconciliation으로 축소 Coco init recovery와의 즉시 중복 제거 중간
P1 Nostr fetchGiftWraps(), subscribeGiftWraps()에 since/limit 적용 relay 과거 이벤트 대량 조회 방지 중간
P2 외부 복구문구 가져오기를 Coco wallet.sweep() 기반으로 재설계 검증 직접 cashu-ts 복구와 limiter 우회 제거 중간
P2 TransferLifecycle 30초 polling을 event 기반 + 장주기 fallback으로 축소 Coco watcher와 중복되는 quote/proof polling 감소 중간
P2 NostrGateway relay scope 분리 temporary relay 누적 후 전체 query/publish되는 문제 완화 중간
P3 cleanAndRecoverStaleMintOps 필요성 재검증 후 제거/축소 Coco mint recovery와 custom recovery 중복 제거 검증 필요

권장 단계별 진행

  1. 계측 먼저: mint URL, endpoint, caller, recovery task id, relay URL, query kind, subscription id를 개발 로그로 묶어 실제 중복을 확인합니다.
  2. 저위험 차단: single-flight/cooldown과 NUT-18 expiry 전달을 먼저 적용합니다.
  3. 민트 요청 통합: direct /v1/info와 external mnemonic restore를 Coco 경유로 옮깁니다.
  4. polling 축소: Coco event가 정상 동작하는 것을 확인한 뒤 Zappi transfer polling 주기를 늘리거나 stuck 대상만 남깁니다.
  5. Nostr 최적화: gift wrap since/limit, relay pool scope, support connect single-flight를 적용합니다.

8. 용어

용어 의미
민트 WSS Cashu mint의 /v1/ws WebSocket입니다. Coco subscription이 사용합니다.
Nostr relay WSS wss://relay... 형태의 Nostr relay 연결입니다. Zappi NostrGateway와 support가 사용합니다.
Polling 상태 변화를 알기 위해 주기적으로 HTTP 요청을 보내는 방식입니다.
Recovery 앱 종료, 네트워크 실패, background 전환 중 멈춘 operation을 재확인하거나 정리하는 작업입니다.
Reconciliation 네트워크 operation은 이미 끝났지만 Zappi 거래내역/상태가 반영되지 않은 경우 로컬 DB를 맞추는 작업입니다.
